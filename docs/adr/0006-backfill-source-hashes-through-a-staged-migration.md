# Backfill source hashes through a staged migration

Existing online SQLite and D1 rows cannot receive their canonical Source Hashes from Koalablog's generic SQL migration because the hash contract is computed in application JavaScript. Gate 3A therefore uses two forward schema stages: `0003` adds Renderer Mode with a Markdown default and temporarily nullable `sourceHash`, normalizes legacy `NULL` content to the blank string, and makes File content non-null, while compatible application code writes canonical hashes for every new or changed Source; during a maintenance window, a revision-guarded JavaScript backfill processes old rows and an audit proves that every stored hash matches the current Renderer Mode and content. Only then does `0004` require `sourceHash` and permit the Renderer UI. Placeholder hashes, permanent `NULL`/blank dual representations, and read-triggered lazy backfills are rejected because they obscure whether the currentness invariant is actually complete.
