# Compile Svelte files entirely in the client

Koalablog builds Svelte files in a lazy browser module Worker, following the current Svelte Playground architecture: the Worker loads one exact Svelte package version, compiles the virtual `App.svelte`, and uses `@rollup/browser` to emit a self-contained IIFE expression with its matching runtime. A `srcdoc` preview iframe executes that bundle through an RPC channel and returns the initial body HTML used as the SEO snapshot. The server never imports `svelte/compiler`, runs Rollup, or executes source; it only saves source, calculates and checks `sourceHash`, persists client-produced artifacts, and serves them. Public delivery embeds the stored IIFE expression in an ES module wrapper exposing an explicit-target mount function, so the page does not need runtime `eval`. HTTPS dependency graphs and Artifact byte size are bounded, and obsolete Worker responses are suppressed by request ID rather than being described as cancelled CPU work. The first client build is heavier than a server compile, but compilation cost remains confined to Svelte editing and rebuild workflows while public visits execute only stored artifacts.
