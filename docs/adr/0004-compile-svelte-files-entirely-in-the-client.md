# Compile Svelte files entirely in the client

Koalablog builds Svelte files in a lazy browser module Worker, following the current Svelte Playground architecture: the deployed application supplies the exact compiler, matching runtime module registry, and `@rollup/browser` as same-origin lazy Worker assets, so browser builds never download the toolchain from npm or another CDN. The Worker compiles the virtual `App.svelte` and emits a self-contained IIFE expression with its matching runtime. A sandboxed opaque-origin `srcdoc` preview iframe executes that bundle through an RPC channel and returns the initial body HTML used as the SEO snapshot; it may make CORS-permitted HTTPS requests but receives no same-origin credentials or private-API proxy, so authenticated runtime data is outside Preview and Snapshot parity. The server never imports `svelte/compiler`, runs Rollup, or executes source; it only saves source, calculates and checks `sourceHash`, persists client-produced artifacts, and serves them. Public delivery embeds the stored IIFE expression in an ES module wrapper exposing an explicit-target mount function, so the page does not need runtime `eval`; once mounted as trusted same-origin application code, the component may make ordinary browser runtime requests. HTTPS dependency graph limits apply to build-time imports, not to public runtime `fetch()`. Artifact byte size remains bounded, and obsolete Worker responses are suppressed by request ID rather than being described as cancelled CPU work. The lazy toolchain increases the Svelte editing download but does not enter ordinary Markdown editing or public delivery bundles. The legacy `/playground`, its server compiler endpoint, and its separate client implementation remain only as a rollback reference until the Worker and Preview gate passes, then are deleted rather than migrated.
