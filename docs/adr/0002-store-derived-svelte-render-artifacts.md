# Store derived render artifacts for Svelte files

Svelte files remain ordinary server-persisted source files even when they are incomplete, invalid, or lack a usable render artifact. A successful client-side render build stores a size-bounded, versioned, self-contained JavaScript bundle, component CSS, and an SEO snapshot derived from the same saved Source Hash, avoiding per-visitor compilation and all server-side compilation or user-code execution. Artifact CSS is served through a checked same-origin stylesheet endpoint rather than interpolated into the Page Shell, so the initial Snapshot is styled without exposing raw Artifact CSS inside an HTML `<style>` context. Saving or importing source never depends on compilation success; browser imports build artifacts afterward on a best-effort basis, while headless `sync-vault` uploads save `.svelte` Source and report that a later browser Rebuild is required without compiling in Node or on the server. The source row stores the canonical hash of Renderer Mode and Source content and updates it atomically with Source Save, so an older artifact becomes non-current immediately. A Current Artifact requires Svelte mode and a matching hash rather than a matching File revision; a preserved Artifact can therefore become current again after an exact Source reversion, while attaching another build replaces the one stored Artifact instead of creating history. A missing or mismatched artifact produces an explicit render failure instead of source exposure, stale output, or a draft/publication state. An otherwise accessible Svelte File without a Current Artifact returns a `503 Service Unavailable` Page Shell with a clear unavailable body and `Cache-Control: no-store`; `404` remains reserved for a missing, unauthorized private, trashed, or purged File. Renderer Mode does not change the existing private-Page login flow: an unauthenticated Page request may continue to the guest-login route, while direct unauthenticated Artifact resource requests return `404`. Public Page HTML is regenerated with `public, no-cache` and has no Artifact-derived ETag in v1 because Page Shell configuration is outside `artifactHash`; only checked public JavaScript and CSS resources use representation-specific strong ETags and may return `304`. Private Pages and resources are always `private, no-store`. Existing artifacts keep their pinned Svelte version until rebuilt. The Page Shell retains Snapshot fallback only through module import and initial mount, removes it after a successful mount, and does not install a global error handler or restore static DOM for later application failures. Artifact requests always recheck current File access and lifecycle state, and trashed or purged Files are unavailable.
